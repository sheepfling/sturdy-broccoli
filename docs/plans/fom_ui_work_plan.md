# FOM UI Work Plan

This document is the execution plan for finishing the FOM workbench UI already
in this repo. It is written to be worked straight through, with clear exit
criteria and verification gates.

The job is not to build a separate frontend product. The job is to turn the
generated HTML workbench into a strong operator console for FOM inspection,
merge triage, validation review, and repo-owned repair flows.

## Objective

The UI is done when an operator can answer these questions from one browser
surface without opening raw XML or JSON first:

1. what is in this family or load set
2. what changed between these two sets
3. why this set failed to merge or validate
4. which member owns the failing declaration
5. what command or repo-owned repair step should happen next

## Current State

The current workbench already has the following in place:

- generated HTML and JSON snapshot output via `./tools/fom-workbench`
- catalog filters and catalog status badges
- browser-saved custom load sets
- ordered load-set builder controls
- candidate mergeability preview in the builder
- conflict workspace with grouped conflict evidence
- inline validation workspace with grouped issue summaries
- inspect and diff panes
- hierarchy-aware object and interaction drill-down
- validation packet links for families and named custom load sets
- symbol jump paths from conflict, validation, search, and diff surfaces
- repo-owned repair suggestions for supported simple datatype conflicts
- guarded datatype edit previews for repo-owned repairs
- copyable command blocks and clearer empty states
- diff defaults that follow the active catalog selection

The implementation slices in this plan are complete. The remaining work is
normal evolution, not planned finish-line work.

The follow-on plan for turning the completed operator console into a polished
application experience lives in:

- [fom_ui_polish_plan.md](fom_ui_polish_plan.md)

## Non-Negotiable Rules

1. The UI stays driven by precomputed repo snapshots. No browser-side XML parsing.
2. Merge and validation truth stays in Python helpers. JavaScript only renders and navigates.
3. Browser-only saved load sets stay non-authoritative until regenerated by the tool.
4. Third-party XML remains read-only.
5. Repo-owned edits stay explicit, previewable, and guarded.
6. Every red state must identify:
   - failure kind
   - failing symbol or location
   - owning member or members
   - exact next step

## Finish Line

The work plan is complete when all of these are true:

1. catalog, inspect, diff, conflict, validation, and search work as one surface
2. conflict and validation issues can jump directly into symbol investigation
3. selected symbol context survives pane changes
4. repo-owned repair suggestions are visible and actionable for supported cases
5. no edit flow silently mutates third-party XML
6. focused workbench tests are green
7. broader FOM suites are green

## Work Sequence

Do the remaining work in this order:

1. symbol drill-down and jump paths
2. guarded repair workspace
3. command bundle polish and operator ergonomics
4. final regression pass across FOM suites

This order is deliberate. Symbol navigation is the missing connective tissue.
Repair flows should land on top of that, not before it.

## Detailed Execution Checklist

### 1. Symbol Drill-Down

Tasks:

- wire a reusable jump helper for symbol-driven navigation
- synchronize:
  - search query
  - selected search result
  - selected tree node
  - inspect pane symbol context
- infer object vs interaction tree target when symbol lineage makes that obvious
- degrade cleanly for datatypes and unresolved names

Definition of done:

- one click from conflict or validation evidence opens the relevant symbol context

### 2. Repair Workspace

Tasks:

- classify supported repairable cases from existing repo-owned edit helpers
- render preview card with:
  - target member
  - target file
  - edit summary
  - exact rerun command
- keep unsupported cases explicit instead of pretending they are editable

Definition of done:

- supported cases show a safe next step, unsupported cases explain why not

### 3. Operator Polish

Tasks:

- tighten labels and status wording
- review panel defaults for the most common failure path
- ensure command blocks are easy to copy and match actual tool usage
- make browser-saved set state obvious when it is not yet materialized by the repo

Definition of done:

- the primary operator path is obvious without documentation open beside it
