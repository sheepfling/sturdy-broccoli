surface.

For the canonical documentation hierarchy, see
[documentation_hierarchy.md](documentation_hierarchy.md).

1. [backend_route_inventory.md](backend_route_inventory.md): exhaustive route inventory and evidence anchors
2. [rti_options_and_test_matrix.md](rti_options_and_test_matrix.md): option inventory and recommended test matrix

## Scope

- Backend columns refer to what a Python federate can rely on through
  `hla2010_rti_runtime_common.create_rti_ambassador(...)`.
- CERTI evidence must distinguish the active default runtime from two named
  baselines:
  - `certi-upstream`: pristine/original CERTI selected by
    `HLA2010_CERTI_UPSTREAM_PREFIX`
  - `certi-patched`: repo-local vendored/patched CERTI selected by
    `HLA2010_CERTI_PATCHED_PREFIX` or the repo-local default
- For the exhaustive route inventory, including `grpc` and `rest` hosted
  variants, see
  [backend_route_inventory.md](backend_route_inventory.md).
- Pitch now has a Docker-backed real-runtime path for exchange/time,
  synchronization, and the basic ownership scenario.
- Pitch negotiated ownership is now known to be bridge-divergent between
  `pitch-jpype` and `pitch-py4j`, not merely "not promotable". See
  [pitch_negotiated_ownership_vendor_bug_2026-06-07.md](../packages/hla2010-rti-pitch-common/docs/evidence/pitch_negotiated_ownership_vendor_bug_2026-06-07.md)
  and
  [diagnostic_summary.md](../analysis/pitch_negotiated_ownership_2026-06-07/diagnostic_summary.md).

Use this file to answer:

- which IEEE 1516.1-2010 (2010 edition) clauses have working behavior now
- where Python RTI is ahead of vendor/runtime paths
- where CERTI or Pitch remains partial beyond the currently promoted overlap

For CERTI-specific caveats, baseline-selection commands, and the distinction
between repo-local patched CERTI and pristine upstream CERTI, see
[certi_runtime_limitations.md](../packages/hla2010-rti-certi/docs/certi_runtime_limitations.md).

Legend:

- `yes`: implemented and exercised by automated tests
- `no`: no backend-neutral implementation or no exercised runtime path in this workspace
- `partial`: implemented, but with narrower semantics or only adapter/unit coverage
- `probe-fails`: service wiring exists, but the local real CERTI runtime fails when exercised
- `blocked`: runtime not currently executable in this workspace

## Matrix

The `CERTI runtime` column currently records active-default evidence. When a
row is sensitive to our local CERTI modifications, promote it to separate
`CERTI upstream` and `CERTI patched` evidence instead of overwriting the shared
column.

Current named CERTI baseline split for the promoted time-query probe,
reproduced through `./tools/certi-easy smoke compare`:

- `certi-upstream`: create/join succeeds, then the first `queryGALT` collapses
  the RTIA path before clause-level time-query semantics can be proved.
- `certi-patched`: the same probe gets through `queryGALT`, `queryLITS`,
  regulation, and constraint setup; `queryGALT` / `queryLITS` return the
  current lookahead after the two-federate regulation/constrained setup, and
  `flushQueueRequest` no longer throws.
- The queued-FQR compare slice now makes the remaining divergence concrete:
  - `certi-upstream`: does not stay alive long enough to finish object setup
    for the queued-FQR scenario
  - `certi-patched`: now drains queued updates in timestamp order and grants at
    the earliest queued timestamp on the Python-facing path
- The same compare route now also carries a negotiated-ownership baseline:
  - `certi-upstream`: the ownership compare probe still fails with an RTI
    internal error before the patched branch semantics can be observed
  - `certi-patched`: the ownership compare probe completes end to end,
    including assumption callback delivery, release request, cancellation
    confirmation, release-response transfer, and final ownership query callback
- The same compare route now also carries release-request branch probes:
  - `certi-upstream`: `deny`, `confirm`, and `ifwanted` all fail before stable
    branch semantics can be observed
  - `certi-patched`: `deny` is distinct, while `confirm` and `ifwanted` expose
    the same effective transfer behavior
  - that patched `confirm` vs `ifwanted` equivalence is also captured by one
    direct profile-comparison test, not only by reading separate branch rows
- If the local session cannot bind/connect loopback TCP on `127.0.0.1`, both
  baseline probes skip before runtime evidence is collected. Treat that as a
  host/session limitation, not a CERTI conformance result.
