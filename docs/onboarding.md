assumes, and where to go next.

## What This Path Assumes

- you want the Python RTI first
- you want package-owned examples and FOM resources
- you want one concrete editing and tracing lane before the broader doc maps
- you want traceability and verification only after the basic example works

## What This Path Avoids

- unpacked evidence packets
- historical source inventories
- archive and provenance indexes
- vendor-runtime troubleshooting until the base Python route is green
- broad package-family reading before one runnable or editable path is clear

## What This Path Gives You

After the first five steps, a human should be able to:

1. run the Python RTI path
2. change one RTI service
3. scaffold one package-owned FOM/federate example
4. trace one method from requirement row to implementation and proof
5. tell which package family they should edit first

If the traced requirement mapping is wrong, the next narrow doc is
[`requirements_edit_one_row.md`](requirements_edit_one_row.md). Start with
[`requirements_crud.md`](requirements_crud.md) when you need the smallest
create/read/update/delete path. The broader
authoring map is secondary reading, not a first-pass front door.
